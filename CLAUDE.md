# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

This repo backs an OKX/Binance MA-cross trading strategy: a Python `freqtrade` strategy/backtest setup (`freqtrade_userdir/`) and a Vue 3 WebUI (`freqtrade-webui/`) that re-implements the backtest/optimize/scan/validate logic client-side in TypeScript so it can run entirely in the browser against live OKX market data.

There is no top-level build — the two halves are independent and not wired together by tooling.

## freqtrade-webui/ (Vue 3 + TypeScript + Vite)

Commands (run from `freqtrade-webui/`):
```
yarn dev       # start dev server on port 3030 (auto-opens browser)
yarn build     # type-check (vue-tsc -b) then vite build
yarn preview   # preview a production build
```
Uses `yarn` (yarn.lock is committed, no package-lock.json). No test runner or lint script is configured in package.json.

### Architecture

The app has **no backend** — it fetches candle data directly from the public OKX REST API in-browser and does all indicator/backtest math in TypeScript. Key flow:

- `App.vue` owns all top-level state (results for each tab, current config) and dispatches the four operations: `handleRunBacktest`, `handleOptimize`, `handleScan`, `handleValidate`. Each populates a result ref and switches `activeTab`.
- `composables/useDataFetch.ts` — fetches OKX candles (`GET https://www.okx.com/api/v5/market/candles`), paginates past OKX's 300-candle-per-request limit via the `after` param, and caches results in a **module-level `globalCache` Map** (shared across all component instances, not per-composable-call). If the OKX request fails, it transparently falls back to `generateMockData()` (deterministic-ish pseudo-random walk) and flips `isRealData` to false — callers should not assume real data was returned.
- `composables/useBacktest.ts` — contains the actual trading logic, independently reimplemented in TS (not calling into the Python freqtrade strategy):
  - `calculateMA` / `calculateADX` — indicators computed manually on the `string[][]` OHLCV arrays (format: `[open, close, low, high, volume]`, all as strings — note close is index 1, not index 3).
  - `runBacktestWithParams` — MA fast/slow crossover + ADX threshold entry, with stop-loss/take-profit and optional short-selling (`enableShort`). This is the single source of truth for backtest math; `OptimizeTab`, `ScanTab`, and `ValidateTab` all call it in loops over parameter grids rather than having separate logic.
  - `calculateSignalScore` — heuristic 0-100 scoring of current signal strength (trend, signal age, ADX, historical win rate/drawdown/return, trade count) used to gate entries in the "validate" (walk-forward) mode and to rank results in "scan" mode.
- Four tabs (`tabs/*.vue`) render results but don't fetch or compute — they're pure display + drill-down UI:
  - `BacktestTab` — single backtest result + equity curve chart (via `ChartPanel.vue`, which wraps `echarts`).
  - `OptimizeTab` — brute-force grid search over `maFast` (4-20) × `maSlow` (fixed set), sorted by total return; can emit `applyParams` back up to `App.vue` to re-run a backtest with chosen params.
  - `ScanTab` — runs the optimize grid search across multiple pairs/timeframes and ranks by signal score.
  - `ValidateTab` — walk-forward validation: slides an optimize-window/predict-window pair across history, re-optimizing params each round and simulating trades only in the unseen predict window (see `handleValidate` in `App.vue` for the round-by-round bookkeeping — this is the most complex code path in the app).
- `types/index.ts` defines shared result/trade shapes used across composables and tabs.

### Important gotchas

- `freqtrade_userdir/webui/index.html` is a **separate, standalone legacy vanilla-JS build** (~3900 lines) of the same UI/logic — it is not generated by `vite build` and is not kept in sync automatically, and is no longer what `deploy.sh` ships.
- `deploy.sh` (repo root) now builds and deploys the **Vue app** (`freqtrade-webui/dist`) to nginx, and builds/(re)starts the `notify-service` backend via PM2 (`premium-notifier`). It requires `notify-service/.env` to already exist (SMTP + optional `HTTPS_PROXY` config) and `pm2` to be installed on the target server. nginx must be configured to reverse-proxy `/api/notify` to the notify-service port, and `freqtrade-webui/.env` (`VITE_NOTIFY_API_BASE`, see `.env.example`) must match that path/host — otherwise the frontend's notification settings tab can't reach the backend.
- OHLCV row format is `[open, close, low, high, volume]` (close before low/high) — easy to mix up with the usual OHLC order.

## freqtrade_userdir/ (Python)

- `config_okx.json` / `config_backtest.json` — freqtrade configs for live/dry-run (OKX exchange, API server on 127.0.0.1:8080) vs. offline backtesting (binance data source, API server disabled). Both are spot trading, dry-run, `stake_amount: 100`, pairs: BTC/ETH/SOL/XRP/DOGE vs USDT.
- `strategies/okx_ma_cross.py` — the actual freqtrade `IStrategy` (MA10/MA30 cross + ADX>20 on 1h candles), run via the standard freqtrade CLI (`freqtrade backtesting`/`freqtrade trade`, not invoked from this repo directly).
- `offline_backtest.py` — a standalone script (not freqtrade-integrated) that reimplements the same MA-cross+ADX logic a third time using `talib`/pandas directly on local `.feather` files under `data/binance/`; run directly with `python offline_backtest.py`. Note it hardcodes a Windows site-packages path and a `f:/work1/k/...` data path — check/adjust these before running.
- Three independent implementations of essentially the same MA-cross strategy exist in this repo (freqtrade strategy class, TS in `useBacktest.ts`, and `offline_backtest.py`) — when changing strategy logic, check whether it needs to be mirrored in the others rather than assuming one file is canonical.
