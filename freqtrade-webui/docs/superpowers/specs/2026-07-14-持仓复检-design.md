# 持仓复检功能设计

日期：2026-07-14
状态：待用户审阅

## 背景与目标

趋势扫描 Tab（`TrendScanTab.vue`）目前只做一次性的品种打分和推荐，用户手动下单后，这个品种就从系统的视野中消失了——没有任何机制帮助用户判断已经买入的仓位后续应该继续持有、减仓还是平仓。

本功能允许用户在扫描结果表格中把某一行标记为「已持仓」，系统记录建仓时的评分快照，此后用户可以在新增的「持仓」Tab 中对这些持仓重新拉取行情、重新打分，并根据评分相对建仓时的变化给出持有/减仓/平仓的建议。

不做自动化交易，只做决策辅助；不接交易所下单接口。

## 方案取舍

评估过 3 种存储/复检方式：

1. **localStorage 本地持久化 + 手动/进入页面时触发复检**（采用）：无需后端，符合本项目"无后端、纯前端"的既有架构；复检由用户主动触发（进入 Tab 或点按钮），不需要常驻定时器或后台任务。
2. **纯内存状态（不持久化）**：实现更简单，但刷新页面后持仓记录全部丢失，不满足"后续持续跟踪"的诉求，放弃。
3. **后端服务持久化 + 定时轮询自动复检**：功能最完整，但需要新增后端和调度机制，与项目当前架构（无后端、OKX API 直连）冲突太大，超出本次需求范围，放弃。

采用方案 1，与项目现有的"前端直接算、无后端"架构保持一致。

## 第一节：数据结构 `Position`

新增 `src/composables/usePositions.ts`，定义：

```typescript
export interface Position {
  id: string                    // `${pair}-${timeframe}-${entryTime}`
  pair: string
  timeframe: string
  direction: 'long' | 'short'
  entryPrice: number
  entryTime: number              // Date.now()
  entryTrendScore: number
  entryGridScore: number
  entryStrategyRecommendation: 'trend' | 'grid' | 'mixed' | 'avoid'
  status: 'open' | 'closed'
  closedTime?: number
  // 复检后写回的最新快照（未复检过时为 undefined）
  lastReview?: PositionReview
}

export interface PositionReview {
  reviewTime: number
  currentPrice: number
  currentTrendScore: number
  currentGridScore: number
  scoreDelta: number             // currentTrendScore - entryTrendScore
  suggestion: 'hold' | 'reduce' | 'close'
  reasons: string[]              // 命中的规则描述，用于表格展示
}
```

持仓记录通过 localStorage 持久化，key 为 `trendscan_positions`，整个 `Position[]` 数组序列化存储。

## 第二节：复检规则引擎 `evaluatePosition`

`usePositions.ts` 中新增：

```typescript
function evaluatePosition(position: Position, fresh: TrendScanResult): PositionReview
```

规则基于**相对建仓时的变化**而非绝对值判断（因为不同品种、不同市场环境下评分的绝对水平本身就有差异，只有相对变化才能反映"这笔仓位是不是在变差"）：

- `scoreDelta = fresh.trendScore - position.entryTrendScore`
- 若方向翻转（`fresh.direction` 与 `position.direction` 相反）→ `suggestion = 'close'`，reason: "趋势方向已反转"
- 若 `scoreDelta <= -30`（评分大幅走弱）→ `suggestion = 'close'`，reason: "趋势评分较建仓时下降超过30分"
- 若 `scoreDelta <= -15` → `suggestion = 'reduce'`，reason: "趋势评分较建仓时下降超过15分"
- 若价格触及建仓时记录的 `stopLossWide`（需在 Position 中额外记录建仓时的止损价，见下方补充字段）→ `suggestion = 'close'`，reason: "价格已触及建仓时的宽止损位"
- 否则 → `suggestion = 'hold'`，reason: "评分与建仓时相比无明显恶化"

> 补充字段：`Position` 需额外记录 `entryStopLossWide: number`（建仓时的宽止损价），用于上述止损触及判断。已并入第一节的数据结构中一并实现。

多条规则同时命中时，`reasons` 数组累加所有命中的原因，`suggestion` 取其中最严重的一个（`close` > `reduce` > `hold`）。

## 第三节：数据获取与复检触发

`PositionsTab.vue` 在以下时机触发复检：

1. **组件挂载时**：对所有 `status === 'open'` 的持仓，逐个调用 `useDataFetch().loadData(pair, timeframe)` 拉取最新K线，再调用 `scoreSymbol()` 重新打分，最后调用 `evaluatePosition()` 得到最新建议，写入该持仓的 `lastReview` 字段（同步回 localStorage）。
2. **手动"重新复检"按钮**：每一行提供按钮，用户可随时对单个持仓重新拉取数据并复检，不必等待重新进入页面。

不做定时自动轮询（后台常驻定时器），复检始终由用户的显式操作触发，与项目"无后端"的定位一致。

## 第四节：在趋势扫描表格中标记持仓

`TrendScanTab.vue` 的「操作」列，在现有"查看K线"按钮旁新增"标记持仓"按钮：

- 点击后调用 `usePositions().addPosition(result)`，用当前这一行的 `TrendScanResult` 构造一条新的 `Position` 记录（`status: 'open'`）。
- 按钮仅在 `direction !== 'neutral'` 且 `!r.insufficientData` 时可用（中性方向的行没有明确的建仓方向，不适合标记持仓）。

**已识别的前置缺口**：`calculateSwingSLTP`（`useTrendScore.ts` 约227行）内部已经计算了 `currentPrice = parseFloat(data[data.length - 1][1])`，但没有把它放进返回的 `SwingSLTPResult`；`scoreSymbol()` 返回的 `TrendScanResult` 也没有暴露 `currentPrice` 字段。标记持仓需要用到建仓时的当前价格，因此需要：

- 在 `SwingSLTPResult` 接口中新增 `currentPrice: number` 字段，并在 `calculateSwingSLTP` 的返回值中包含它。
- 在 `TrendScanResult` 接口（`src/types/index.ts`）中新增 `currentPrice: number` 字段，并在 `scoreSymbol()` 的返回对象中透出 `sltp.currentPrice`。

这个改动是本功能实现的必要前提，将在实现计划中作为第一步。

## 第五节：持仓 Tab 界面设计

新增 `src/tabs/PositionsTab.vue`，通过 `App.vue` 新增一个 Tab 项接入（Tab 名称："持仓"）。

表格列：

| 列 | 说明 |
|---|---|
| 交易对 | `position.pair` |
| 周期 | `position.timeframe` |
| 方向 | 做多/做空 |
| 建仓价 | `position.entryPrice` |
| 建仓时间 | 格式化后的 `position.entryTime` |
| 建仓分 | `position.entryTrendScore` |
| 当前分 | `lastReview.currentTrendScore`，未复检时显示"未复检" |
| 分数变化 | `lastReview.scoreDelta`，正数绿色/负数红色 |
| 建议 | `lastReview.suggestion`（持有/减仓/平仓），用颜色区分（绿/黄/红） |
| 命中原因 | `lastReview.reasons.join('；')` |
| 操作 | "重新复检" 按钮 + "平仓" 按钮 |

**平仓与历史行为**：点击"平仓"按钮后，将该持仓的 `status` 置为 `'closed'`，写回 localStorage；行**不删除**，而是保留在列表中，通过表格上方的筛选按钮（"全部 / 持仓中 / 已平仓"）过滤显示，用于事后回顾历史持仓的表现。

## 范围之外（本次不做）

- 不接交易所下单/查询持仓接口，仓位数量、盈亏金额均不计算，只做"建仓价 vs 当前价"和"评分变化"的定性判断。
- 不做定时自动轮询复检。
- 不做多标的组合层面的风险分析（比如仓位相关性、总敞口）。
